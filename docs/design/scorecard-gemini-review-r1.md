3개 설계안에 대한 교차 리뷰 및 최종 통합 스코어카드 설계를 위한 판단을 정리합니다.

## 1. 각 안의 핵심 강점과 결정적 약점

### [A] Claude 설계안
*   **강점**: **"게이트"와 "점수"의 분리**, 그리고 **LLM의 이진/범주형 판정(Y/P/N) 강제**. 가설 품질 평가에서 가장 골칫거리인 'LLM의 점수 진동(Jitter)'을 원천 차단하여 재현성을 극대화한 최고의 엔지니어링 접근입니다. '피드백 소진' 기반의 정체 감지 논리도 매우 탁월합니다.
*   **약점**: 통과 기준(78점 + 다중 AND 조건)이 지나치게 빡빡합니다. 실제 현업 데이터 분석가가 작성한 초기 가설도 이 기준에 걸려 무한 루프나 조기 종료를 겪을 확률이 높습니다.

### [B] GPT 설계안
*   **강점**: 완벽주의를 버린 **"조건부 통과(Acceptable with Caveat)"** 등급 도입. UX 관점에서 루프 피로도를 낮추고 사용자에게 맥락을 넘기는 실용적인 접근입니다. "Information Gain"과 "반증가능성"을 핵심 기준으로 삼은 철학이 훌륭합니다.
*   **약점**: 메커니즘 구체성을 '15자 이상'이라는 글자 수로 판정하는 것은 LLM이 의미 없는 수식어로 우회(Gaming)하기 너무 쉽습니다. LLM에게 1~5점 척도 채점을 맡긴 점은 A안과 비교해 재현성 측면에서 명백한 하수입니다.

### [C] Gemini 설계안 (나의 설계안 자체 비판)
*   **강점**: 피드백을 JSON 구조화하여 주입하는 방식이 가장 깔끔하며, LLM(sharpener)이 정확히 무엇을 고쳐야 할지 인지하기 좋습니다.
*   **결정적 약점 (자기 비판)**: 3개 안 중 **가장 퇴보한 LLM 채점 방식(0~30점 자유 채점)**을 제안했습니다. 이로 인해 점수 진동이 필연적으로 발생하는데, 무한루프 방지책으로 "점수 상승폭 5점 이하 시 종료(Delta Check)"를 넣은 것은 논리적 모순입니다. LLM의 무작위 진동에 의해 루프가 헛돌거나 조기 이탈할 위험이 매우 큽니다. 또한 필수 데이터 유무(구조적 완전성)에 25점이나 부여한 것은 점수 인플레이션을 유발합니다.

## 2. 통합 스코어카드에 반드시 들어가야 할 아이디어

1.  **점수 산출에서 LLM 배제 / Y·N·P 이진 판정화 (from A)**
    *   LLM은 오직 '반증 시나리오 도출 가능 여부', '인과 고리 단절 여부'만 판정하고, 점수(숫자)는 파이썬 결정론 룰이 매핑해야 합니다.
2.  **게이트(결격 사유)와 스코어의 분리 (from A)**
    *   측정 불가능하거나 반증 불가능한 가설은 총점이 90점이라도 차단(Fail)해야 합니다.
3.  **조건부 통과 (Acceptable with Caveat) 등급 (from B)**
    *   완벽하지 않아도 핵심 게이트만 넘으면 통과시키되, `bias_screener`나 사용자 UI에 "교란 변수 통제가 부족함" 등의 경고(Caveat)를 전달하여 진행을 막지 않아야 합니다.
4.  **피드백 소진 기반 정체 감지 (from A)**
    *   점수 변화폭(Delta)이 아니라, "동일한 차원이 2회 연속 미달인가?"를 기준으로 루프 정체를 감지해야 합니다.

## 3. 버려야 할 것 / 충돌하는 설계 결정과 판단

*   **버려야 할 것 1: LLM의 N점 척도 채점 (B의 5점, C의 30점)**
    *   [판단] 전면 폐기. 프롬프트를 아무리 깎아도 LLM은 매 턴 숫자를 바꿉니다. A안의 `Literal["Y", "P", "N"]` 구조화 출력이 유일한 정답입니다.
*   **버려야 할 것 2: 길이 기반 룰 (B의 15자, D5의 가정 N개 등)**
    *   [판단] LLM은 글자 수나 리스트 개수를 채우기 위해 동어반복을 만들어냅니다. 개수 룰을 쓰더라도 A안처럼 동어반복(성공, 개선 등) 차단 룰을 반드시 결합해야 합니다.
*   **충돌 지점: 루프 정체 시 액션 (A: Best-so-far 반환 vs B: Expander로 롤백)**
    *   [판단] **B의 Expander 롤백은 위험합니다.** 세션 컨텍스트가 너무 커지고, 완전히 다른 가설로 튀어버리면 사용자 의도(원래 검증하고 싶었던 것)가 훼손됩니다. 정체 시에는 A안처럼 **"Best-so-far + 미해결 Caveat"**를 들고 다음 단계로 전진(Soft Pass)하는 것이 UX상 안전합니다.

---

## 4. 미해결 쟁점 (통합 과정에서 풀어야 할 숙제)

1.  **멀티모델 호환성 (Enum 강제력)**
    *   A안의 핵심은 LLM이 정해진 Enum(Y/N/P)만 뱉는다는 가정입니다. 현재 `ab-lens`는 Claude, OpenRouter, Gemini 등 다양한 Provider를 지원해야 하므로, Structured Output(또는 JSON Schema) 적용 시 각 API별 엄격한 Enum 강제가 동일하게 작동하는지 검증이 필요합니다. 실패 시 강력한 파이썬 정규식 폴백(Fallback)이 필요합니다.
2.  **부분 점수(P)에 대한 피드백 생성 한계**
    *   LLM이 "인과 고리가 끊겼다(P)"고 판정했을 때, 구체적으로 *어디가* 끊겼는지(Mechanism gap) 짧은 텍스트로 잘 추출해내어 sharpener에게 주입할 수 있느냐가 루프 수렴 속도를 결정합니다.
3.  **임계값(Threshold) 튜닝의 문제**
    *   A안의 78점, B안의 70점 모두 직관에 의한 매직 넘버입니다. 초기에는 점수(숫자)보다는 **"Disqualifier(결격) 0개 + 1개 이하의 Minor 결함 = 통과"** 같은 규칙 기반 등급 산정이 더 견고할 수 있습니다.
